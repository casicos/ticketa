# Ticketa — 추가 디자인 의뢰 브리핑 (2026-05-11)

기존 디자인 번들 `ticketa-design-system` 의 톤·컴포넌트·컬러 토큰 그대로 사용.
모든 시안은 데스크탑(1216px 컨테이너) + 모바일(390px) 동일 흐름이면 충분.

| Priority | 묶음                                                              | 비고                                          |

| **P1**   | D. `/agent/store` 등록 / E. 판매 등록 모달 / F. 위탁 입고 승인 큐 | 시나리오 [4]                                  |

## A. `/admin/intake` 송장 입력 폼 (P0)

**맥락**: 어드민이 매물을 구매자에게 발송할 때 사용. `verified → shipped` 트랜지션의 입력 폼.

**필드**

- 택배사 (select, 필수)
  - 우체국 / CJ대한통운 / 한진 / 롯데 / 로젠 / 편의점-CU / 편의점-GS25 / 편의점-이마트24 / 편의점-세븐일레븐 / 기타
- 송장번호 (input, 필수, 숫자/하이픈)
- 발송 메모 (textarea, 선택)

**액션**: "발송 처리" 버튼 → 매물이 `shipped` 로 전환됨.

**디자인 요구**

- 매물 행에서 "발송 처리" 클릭 시 인라인 폼 펼침 또는 다이얼로그
- 택배사 select 는 한국 택배사 브랜드 컬러 활용 가능 (옵션)
- 발송 후 송장 정보가 매물 상세에도 표기됨 — 그 카드 디자인도 함께

---

## B. listing `[인증]` 배지 (P0)

**맥락**: 사전검수 완료된 매물 (P2P + admin 검수 끝난 것 OR 에이전트 매물). 카탈로그/상세에 배지로 표시.

**조건**

- 카탈로그 카드에 한 곳, 상세 페이지에 한 곳
- 색: 신뢰감 있는 톤 (그린 계열 추천 — `#1F6B43` 같은)
- 기존 카드의 "검수완료" 표기와 의미 통일 필요 — `[인증]` 으로 대체 또는 통합

**의미**

- "관리자가 실물 검수를 마쳐서 매입 즉시 발송 가능한 매물" 임을 사용자에게 시각적으로 명확하게.
- 비-인증 매물에는 배지 없음 (현재 P2P 흐름 그대로 진행).

---

## C. 카탈로그 카드/상세에 상점명 노출 (P0)

**맥락**: 에이전트 매물은 판매자가 상점명으로 표기됨 (예: "컬쳐 에이전시"). P2P 매물은 "판매자 ABC1234" 같은 익명 코드.

**디자인 요구**

- listing-card 안에서 상점명이 들어갈 위치 (DeptMark 옆? 가격 위? 카드 하단?)
- 에이전트 매물임이 시각적으로 한 번에 식별되는 톤 (예: 골드 컬러 작은 라벨 + 상점명)
- 상세 페이지에서는 상점명 클릭 → 상점 페이지(준비중) 로의 호버/링크 패턴

**참고**: 기존 디자인 번들의 `screens-trade.jsx` 의 listing 카드를 출발점으로.

---

## D. `/agent/store` 상점 등록 폼 (P1)

**맥락**: 에이전트 승급 후 상점명/소개를 등록. 1인 1상점.

**필드**

- 상점명 (input, 필수, 2~20자, 중복 체크)
- 한 줄 소개 (input, 선택, ~60자)
- (선택 — 다음 단계) 로고/컬러

**디자인 요구**

- `/agent` 사이드바의 "상점 브랜드" 메뉴 진입점
- 등록되지 않은 상태와 등록된 상태(편집) 두 화면
- 미리보기 카드 (사용자 카탈로그에서 어떻게 보이는지)

---

## E. 에이전트 판매 등록 모달 (P1)

**맥락**: `/agent/inventory` 에서 보유 재고 한 건을 클릭 → 판매 등록 모달. inventory 에서 끌어와 listing 으로 만드는 흐름.

**필드**

- SKU (read-only, 재고 row 에서 자동 입력)
- 위탁가 / 위탁 수량 (read-only, 표시만)
- 판매 수량 (input, 1~보유수량)
- 판매가 (input, 위탁가 ± 안내)
- (선택) 판매 시작/종료일

**액션**: "판매 등록" → 카탈로그에 즉시 노출됨 ([인증] 배지 자동).

**디자인 요구**

- 모달 또는 슬라이드 시트
- 위탁가가 위에, 판매가가 아래로 시각 대비
- 등록 후 inventory 표에서 "판매중 N장 / 보유 M장" 형태로 분할 표기

---

## F. `/admin` 위탁 입고 승인 큐 (P1)

**맥락**: 에이전트가 위탁한 실물을 어드민이 수령 → 검증 → inventory 적재.

**입력**

- 큐 목록 (에이전트, SKU, 수량, 위탁가, 송부일)
- 각 행에 "수령 확인" / "검증 완료 + 적재" 액션
- 적재 시 inventory 행 생성됨

**디자인 요구**

- 기존 `screens-admin.jsx::DesktopAdminListings` 톤 그대로
- 수령 vs 적재 두 단계로 분리 (수령 후 검증 단계 표시)
- 적재 후 에이전트에게 자동 알림

## G. 선물 흐름 전체 (P2 — 양 많음)

### G-1. 선물 보내기 모달

**맥락**: `/catalog/[id]` 에이전트 매물 상세에서 "선물하기" 클릭.

**필드**

- 받는 사람 닉네임 (input + 자동완성, 사용자 검색 결과 카드 형태)
- 수량 (input)
- 메시지 (textarea, 선택, ~200자)
- 결제 요약: 단가 × 수량 = 합계, 마일리지 차감 안내

**액션**: "선물 보내기" → 마일리지 차감 + 수령자에게 알림.

---

### G-2. `/account/gift` 받은 선물 inbox

**맥락**: 마이룸 사이드바 "선물 상품권" 메뉴 진입.

**컴포넌트**

- 탭: 받은 선물 / 보낸 선물
- 받은 선물 카드: 보낸 사람 닉네임, 매물 정보, 수량, 메시지, 상태(미수령/수령완료/환불됨/만료)
- 미수령 카드에 액션 버튼 2개: "마일리지로 받기" / "현물 배송 받기"
- 7일 카운트다운 표기

---

### G-3. 받은 선물 액션 시트

**맥락**: G-2 카드의 액션 버튼 클릭 시.

**두 가지 액션**

- **마일리지로 받기**:
  - 안내 톤이 중요: "보낸 사람에게는 알려지지 않아요" 명시
  - 합계 × 수량 만큼 마일리지 즉시 충전됨
  - "이 선택은 되돌릴 수 없어요" 확인
- **현물 배송 받기**:
  - 배송지 선택 / 신규 등록
  - 받은 후 배송 단계 진행됨 안내

---

### G-4. `/account/gift` 보낸 선물 outbox

**맥락**: 동일 페이지 다른 탭.

**카드**

- 받는 사람 닉네임, 매물 정보, 수량, 상태
- 미수령 상태에는 "환불 요청" 액션
- 수령 완료 시 어떤 방식으로 받았는지 표기 (단, 마일리지 변환 사실은 마스킹 — "수령 완료" 라고만)

---

### G-5. 배송지 등록 / 선택 시트

**맥락**: G-3 현물 배송 선택 시 처음이면 등록, 이후엔 선택.

**필드**

- 받는 사람 이름 (자동: 본인)
- 휴대폰 번호 (자동: 본인)
- 우편번호 + 도로명 주소 (다음 우편번호 API 또는 단순 input)
- 상세 주소
- 기본 배송지 토글

**컴포넌트**

- 등록된 배송지 리스트 (radio select)
- "+ 신규 배송지" 버튼 → 등록 폼

---

## 톤 / 컬러 / 가이드라인

- 기존 번들 그대로: `--ticketa-blue-*` / `--ticketa-gold-*` / `--warm-*`
- 마이룸/어드민 폰트/사이즈 그대로
- 알림 카피는 한국어, 사용자 대면 라벨은 짧게
- 금액은 `1,234,567원` 또는 `M` 표기. K/만 단위 영문 표기 금지
- 배지는 작게, 라벨은 명확하게

## 전달용 짧은 버전 (디자인 시안 요청 한 줄씩)

P0:

- (A) `/admin/intake` 발송 처리 폼: 택배사 select + 송장번호 + 발송 메모
- (B) listing-card 와 상세 페이지의 `[인증]` 배지 위치/색
- (C) 카탈로그에서 에이전트 매물 상점명 노출 위치 + 에이전트 매물 시각 식별

P1:

- (D) `/agent/store` 상점명·소개 등록 폼 (등록전/등록후 두 화면)
- (E) `/agent/inventory` 의 "판매 등록" 모달 (위탁가 표시 + 판매가 입력)
- (F) `/admin` 위탁 입고 승인 큐 + 적재 액션

P2 (선물 흐름 — 양 많음):

- (G-1) 매물 상세의 "선물 보내기" 모달 (닉네임 검색 + 메시지 + 결제 요약)
- (G-2) `/account/gift` 받은 선물 inbox (탭, 카드, 카운트다운)
- (G-3) 받은 선물 액션 시트 ("마일리지로 받기" silent 안내 + "현물 배송 받기")
- (G-4) `/account/gift` 보낸 선물 outbox (환불 액션, 수령 완료 마스킹)
- (G-5) 배송지 등록/선택 시트
